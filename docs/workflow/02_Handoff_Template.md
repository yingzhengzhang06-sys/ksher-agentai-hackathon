# 跨角色交接模板

## 1. 工程师完成报告模板

```md
## 工程师完成报告

- 任务编号：
- 任务名称：
- 执行角色：
- 完成内容：
- 修改文件：
- 未修改范围声明：
- 测试命令：
- 测试结果：
- 风险 / 阻塞：
- 下一建议角色：
- 下一建议任务：
```

字段要求：

- 修改文件必须列出具体路径。
- 未修改范围声明必须明确没有修改 `.env`、密钥、Token、生产配置和任务单外文件。
- 测试失败时不得进入下一阶段。
- 存在阻塞时必须说明责任角色和处理建议。

## 2. 架构师审核报告模板

```md
## 架构师审核报告

- 审核对象：
- 审核结论：通过 / 有条件通过 / 不通过
- 技术一致性检查：
- MVP 边界检查：
- 风险项：
- 需返工项：
- 下一步建议：
```

字段要求：

- 技术一致性检查必须对照 PRD、Tech Design 和 Engineering Subtasks。
- MVP 边界检查必须确认没有新增超范围功能。
- 有条件通过或不通过时必须列出返工项和验收标准。

## 3. QA 验收报告模板

```md
## QA 验收报告

- 验收范围：
- 执行用例：
- 通过用例：
- 失败用例：
- 缺陷编号：
- 截图路径：
- 责任角色：
- 是否允许进入产品门禁：
```

字段要求：

- 每个失败用例必须关联缺陷编号。
- 每个缺陷必须包含复现步骤和责任角色。
- 截图路径必须可追溯。
- QA 未通过时不得进入产品负责人最终门禁。

## 4. 产品负责人门禁报告模板

```md
## 产品负责人门禁报告

- 门禁对象：
- 结论：通过 / 有条件通过 / 不通过
- 用户价值判断：
- 范围控制判断：
- 验收结果判断：
- 是否允许上线 / 演示 / 归档：
```

字段要求：

- 结论必须基于事实源文档和 QA 验收结果。
- 不允许因个人审美扩大 MVP。
- 有条件通过必须说明条件和责任角色。
