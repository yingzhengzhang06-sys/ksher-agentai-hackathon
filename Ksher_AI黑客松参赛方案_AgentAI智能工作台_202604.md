## 一、问题的本质：不是效率问题，是成本结构问题

### 跨境支付渠道扩张的结构性死结

Ksher正在建设「城市代理商」渠道体系，模式是 **"一人公司"** — 代理商以个人身份运营，帮Ksher拓展本地跨境支付商家。

但渠道负责人面对一个**结构性死结**：

```
现实：渠道团队 = 你 + 有限的团队成员
      ↓
每新增1个代理商，你需要投入：
  ├── 21天孵化培训（话术/产品/异议处理）
  ├── 持续物料支持（方案/海报/PPT/内容）
  ├── 7×24产品答疑（40+文档的知识量）
  └── 一对一陪跑拓客
      ↓
你的时间是线性的，代理商是要规模化的
      ↓
5个代理商 → 你能亲自带
20个代理商 → 你带不过来
100个代理商 → 不可能

▸ 死结：渠道规模化的天花板 = 你的时间上限
```

**这不是"慢"的问题，是"不可能"的问题。** 传统方式下，渠道规模和赋能成本线性绑定，永远无法突破。

### 双战场的挑战加剧了这个死结

代理商需要在两个战场同时作战，对赋能的要求更复杂：

| 战场 | 对手 | 获客逻辑 | 赋能要求 |
|------|------|---------|---------|
| **增量战场**：从银行抢客户 | 银行（占90%跨境支付量） | "帮你算一笔你没算过的账"——用数据撬动惰性 | 需要精确的成本对比分析能力 |
| **存量战场**：从竞品抢客户 | PingPong/万里汇/XTransfer | 差异化切入——东南亚本地牌照+更低费率+更好服务 | 需要竞品对比+定制化方案能力 |

两个战场意味着代理商需要**两套完全不同的话术体系、分析工具和物料**。传统一对一赋能根本覆盖不了。

### 当前赋能瓶颈（真实现状）

| 赋能环节 | 现状 | 痛点 |
|---------|------|------|
| 话术培训 | 渠道团队亲自带 | 每个代理商投入21天，无法并行 |
| 客户方案 | 渠道团队协助制作 | 半天做一份，5个代理商同时要就排不过来 |
| 物料制作 | 依赖产品/设计排期 | 等1周出一张海报，代理商等不起 |
| 产品答疑 | 微信群实时回复 | 40+文档的知识量，你不可能24小时在线 |
| 异议处理 | 社群分享+复盘 | 实战中遇到新异议只能事后补课 |
| 成本对比 | 手动计算 | 每个客户情况不同，一对一算太耗时 |
| 提案PPT/海报 | 模板+手动改 | 不专业，代理商也不会改 |

> **核心矛盾**：你一个人（+有限团队）要完成市场+销售+培训+客服+数据+设计6个角色的工作产出，还要同时支持越来越多的代理商。这不是能力问题，是物理极限。

## 二、解决方案：不是做工具，是改变渠道扩张的成本结构

### 核心洞察（蒂尔视角）

> **大平台（PingPong/万里汇）的利润结构决定了它们只能服务大代理+大商家。中小代理+中小商家这个市场，它们不是不想做，是做不划算。**
>
> **AgentAI让Ksher的代理赋能边际成本趋零——这意味着Ksher可以经济地服务无限多的代理商，包括大平台放弃的那些人。这不是效率优化，是开辟了一个竞品成本结构无法覆盖的市场。**

```
Before：
  你（渠道负责人）
    ├── 亲自培训代理商A（投入21天）
    ├── 亲自培训代理商B（再投入21天）
    ├── 同时给A做方案、回答B的问题、帮C做海报…
    └── 天花板：你能同时支持的代理商 ≤ 5个

After：
  AgentAI 智能工作台
    ├── 代理商A 自助获取全套作战武器（话术+方案+物料+知识）
    ├── 代理商B 同时也在自助使用
    ├── 代理商C…D…E…第100个代理商…
    └── 你的时间从"做执行"释放为"做决策"——渠道策略、大客户攻关、代理商筛选

成本结构改变：
  传统：赋能成本 ∝ 代理商数量（线性）
  AI后：赋能成本 ≈ 常数（API费用 ~¥2,000-5,000/月）
```

### 产品定位

**Ksher AgentAI 智能工作台** = 你的渠道赋能分身

不是7个独立的AI小工具打包，而是一个**以"客户拜访"为核心的统一作战系统**。

## 四、第一层：一键备战（核心体验）

### 「一键备战」全流程

代理商只需**一次输入**，AI自动编排4个引擎协同工作：

```
输入（一次）：
  客户公司 + 行业 + 目标国 + 月流水 + 当前收款方式 + 痛点

                    ┌──────────────┐
                    │  AI 编排中枢   │
                    └──────┬───────┘
            ┌──────┬──────┼──────┬──────┐
            ▼      ▼      ▼      ▼      ▼
         话术引擎  成本引擎  方案引擎  异议引擎  战场判断
            │      │      │      │      │
            ▼      ▼      ▼      ▼      ▼

输出「作战包」（4件武器，一次生成）：

  ⚔️ 武器一：定制话术
    ├── 30秒电梯话术（适配对应战场）
    ├── 3分钟完整讲解
    └── 微信跟进话术（首次添加+后续跟进）

  ⚔️ 武器二：成本对比报告
    ├── 5项总成本全景分析
    │   ├── ① 显性手续费
    │   ├── ② 汇率损失（汇率差×流水）
    │   ├── ③ 资金时间成本（延迟天数×日利率×流水）
    │   ├── ④ 多平台管理成本
    │   └── ⑤ 合规风险成本
    ├── 可视化对比图表
    └── 结论："切换到Ksher，年省 ¥XX"

  ⚔️ 武器三：客户定制方案
    ├── 行业洞察 → 痛点诊断 → 解决方案
    ├── 产品推荐（自动匹配国家/币种/链路）
    ├── 费率优势（引用成本引擎数据）
    ├── 合规保障 + 接入流程
    └── 可下载 Markdown / 一键转PPT

  ⚔️ 武器四：异议预判
    ├── Top 3 最可能的客户异议（基于画像预测）
    ├── 每个异议3种回复策略（直接型/共情型/引导型）
    └── [进入模拟练习] 按钮 → AI扮演客户对练
```

### 双战场智能适配

系统根据客户「当前收款方式」自动判断战场，调整全部输出策略：

| 判断条件 | 战场 | 话术策略 | 成本计算重点 | 方案重心 |
|---------|------|---------|------------|---------|
| 当前用银行电汇 | 增量战场（从银行抢） | "帮你算一笔没算过的账" | 银行隐性成本（汇损+时间成本+固定手续费） | 强调"降维打击"——费率/速度/服务全面碾压 |
| 当前用PingPong等 | 存量战场（从竞品抢） | "东南亚这块，我们有硬差异" | 竞品对比（费率差+汇率差+到账速度差） | 强调差异化——东南亚本地牌照+本地化服务 |
| 当前未选定/新商家 | 教育战场（抢先教育） | "跨境收款有3个坑，你知道几个" | 行业全景分析（所有渠道对比） | 强调选择标准——教客户怎么选，引导选Ksher |

> **惊艳点**：同一个代理商，输入一个银行客户和一个PingPong客户，系统自动切换整套策略。评审现场可对比演示。

**解决**：代理商不会写文案、不会做设计、朋友圈不知道发什么

**四大内容能力**：

| 短视频口播稿 | 时长+主题 | 脚本+画面建议+字幕文本 | 2小时→2分钟 |

### 5.2 🧠 产品知识库智能问答

**解决**：代理商被客户问住就丢单（3大业务×6国×多特色产品，40+文档）

```
代理商随时提问 → AI基于Ksher完整知识库秒答 + 来源引用 + 延伸建议

"泰国B2C电商收款流程？" → 精准答案 + 链路图
"Ksher和PingPong区别？" → 6维度竞品对比
"客户入账失败怎么办？" → 排查步骤 + 处理流程
"POBO退税怎么操作？"  → 产品详解 + 适用场景
```

知识库覆盖：基础知识8篇 + B2C 5国 + B2B 7账户 + 服务贸易6国 + 特色产品5篇 + 竞品分析6篇 + FAQ（共40+文档）

#### 三层知识融合机制

> **核心问题**：代理商的问题不会只停留在知识库能覆盖的范围内。当客户问"泰国电商市场2026年增长趋势如何？Ksher有什么优势？"——知识库有Ksher产品数据，但没有行业趋势数据。系统需要**智能融合多层知识**，而不是只复读知识库。

**三层知识优先级体系**：

```
┌─────────────────────────────────────────────────────────┐
│                                                           │
│  第一层【最高优先】Ksher产品知识库                          │
│  费率/流程/牌照/国家产品卡/竞品对比/FAQ                     │
│  → 涉及Ksher具体产品时，必须以知识库为准                    │
│  → 标注"根据Ksher产品资料"                                 │
│                                                           │
│  第二层【补充】Claude模型自身知识                            │
│  行业趋势/市场数据/电商平台规则/各国政策/通用商业知识          │
│  → 知识库未覆盖的行业背景、市场数据自动补充                   │
│  → 标注"根据行业公开信息"                                   │
│                                                           │
│  第三层【融合】综合推理生成                                  │
│  结合知识库的Ksher优势 + 自身知识的行业背景 → 生成完整回答     │
│  → 既有精准的产品数据，又有说服力的行业背景                   │
│                                                           │
│  冲突规则：知识库 > 模型自身知识（内部数据以知识库为准）       │
│  不确定时：明确标注"此信息需与Ksher内部确认"                  │
│                                                           │
└─────────────────────────────────────────────────────────┘
```

**效果对比**：

| 问题 | 无知识库（纯Claude） | 有知识库但不融合 | 三层知识融合 |
|------|-------------------|----------------|------------|
| "泰国B2C费率多少？" | 不知道/瞎猜 | ✅ 精准回答（知识库） | ✅ 精准回答 |
| "泰国电商市场趋势？" | ✅ 能答（通用知识） | ❌ 回答"知识库中无此信息" | ✅ 行业趋势 + Ksher泰国优势 |
| "为什么选Ksher做泰国？" | 泛泛而谈 | 只列产品参数 | ✅ **"泰国是东南亚第二大电商市场（行业知识），Ksher在泰国有SCB本地账户+秒到宝T+0到账（知识库），相比PingPong需中转到账T+2（竞品知识库），资金周转优势明显"** |

> **惊艳点**：这不是简单的"查手册"式问答，而是一个真正理解跨境支付行业的AI顾问——既能精准引用Ksher产品数据，又能结合行业背景给出有说服力的完整回答。代理商得到的不是"知识库摘抄"，而是"专家级分析"。

#### V1→V2 知识能力演进

| 维度 | V1 Demo | V2 生产版 |
|------|---------|----------|
| 知识库 | Prompt注入（40+文档） | 向量数据库RAG（动态检索） |

> V1阶段不加联网搜索——Claude自身知识已覆盖到2025年中的行业信息，结合40+文档的Ksher知识库，能覆盖代理商95%以上的问题。V2按需加入联网搜索补充实时数据。

### 5.3 🎨 智能海报+PPT生成

## 六、第三层：学习飞轮（真正的壁垒）

### 克里斯坦森洞察：竞品能复制工具，复制不了数据

> Streamlit + Claude API，任何公司一周能复制。
> 但**"哪些话术在哪个行业转化率最高""哪些异议最难处理""银行客户vs竞品客户哪个签约周期短"** — 这些数据来自Ksher代理商的实战积累，竞品没有这个生态，就没有这个数据。

### 数据闭环机制

```
代理商使用「一键备战」生成作战包
        │
        ▼
代理商拜访客户
        │
        ▼
拜访后标记结果（简单2步）：
  ├── 结果：✅签约成功 / ⏳持续跟进 / ❌放弃
```

### 知识库动态管理（保持AI回答与产品同步更新）

> **核心问题**：Ksher的产品在持续迭代——新增国家、调整费率、上线新功能、变更开户流程。如果知识库不更新，AI的回答就会过时，代理商拿着旧信息去见客户，比不用AI更危险。

**因此系统必须提供知识库管理入口**，让渠道负责人（你）能随时更新产品知识，AI的回答实时跟着变。

#### 知识库更新场景

| 触发事件 | 需要更新的内容 | 影响的Agent | 紧急程度 |
|---------|-------------|-----------|---------|
| 费率调整 | fee_structure.json + 对应国家产品卡 | Cost/Proposal/Knowledge | 🔴 立即（错误费率=丢单） |
| 新增国家/币种 | 新增国家产品卡Markdown | Proposal/Knowledge | 🟡 1周内 |
| 开户流程变更 | 对应业务介绍文档 | Knowledge/Proposal | 🟡 1周内 |
| 新特色产品上线 | 新增products/目录文档 | Proposal/Knowledge/Speech | 🟡 1周内 |
| 竞品信息更新 | competitors/对应竞品文档 | Objection/Cost/Knowledge | 🟢 按需 |
| 新增FAQ/常见问题 | operations/FAQ文档 | Knowledge | 🟢 按需 |

## 七、技术架构

### 两版策略

| 维度 | V1 黑客松Demo | V2 生产版 |
|------|--------------|----------|
| 技术栈 | Python + Streamlit + 多模型API | Next.js + 多模型API |
| AI引擎 | Kimi K2.5（主力） + Claude Sonnet 4.6（精准任务） | 同左 + 模型路由优化 |
| 知识库 | 本地Markdown注入Prompt | 向量数据库RAG |
| 数据闭环 | 本地JSON模拟 | 数据库持久化 + 分析引擎 |
| 部署 | 本地运行/Streamlit Cloud | 云服务器 + CDN |
| 开发周期 | 1周 | 1-2个月 |

### 多模型协同策略

> **核心思路**：不是所有Agent都需要最贵的模型。创意型任务（话术/内容/设计）用Kimi K2.5（便宜+中文强），精准型任务（成本计算/方案/知识问答）用Claude Sonnet 4.6（推理准确+不编造）。开发阶段用Opus 4.6写代码（只花一次钱）。

#### 三个模型的定位

| 模型 | 定位 | 成本 | 用在哪 |
|------|------|------|--------|
| **Kimi K2.5** | 中文创作主力 | 较低（输入¥4/MTok, 输出¥16-21/MTok） | 运行时80%调用（话术/内容/设计/异议） |
| **Claude Sonnet 4.6** | 精准推理担当 | 中（输入$3/MTok, 输出$15/MTok） | 运行时20%调用（成本计算/方案/知识问答） |
| **Claude Opus 4.6** | 架构师 | 最高 | 仅开发阶段（写代码/搭架构），不用于运行时 |

#### Agent → 模型映射

| Agent | 分配模型 | 为什么用这个模型 |
|-------|---------|---------------|
| SpeechAgent（话术） | **Kimi K2.5** | 创意型中文生成，Kimi中文能力极强，成本最低 |
| ContentAgent（内容） | **Kimi K2.5** | 朋友圈/小红书/视频脚本，纯中文创作，量大 |
| DesignAgent（设计） | **Kimi K2.5** | 海报文案/PPT文字内容，创意型 |
| ObjectionAgent（异议） | **Kimi K2.5** | 对话式回复生成，中文对话自然 |
| CostAgent（成本） | **Sonnet 4.6** | 需要精确解读数学计算结果，不能出错 |
| ProposalAgent（方案） | **Sonnet 4.6** | 结构化方案+准确引用知识库数据，严谨性要求高 |
| KnowledgeAgent（知识） | **Sonnet 4.6** | 必须精准匹配知识库+绝不编造信息 |

#### 成本对比

| 方案 | 月成本（20个代理商） | 年成本 |
|------|-------------------|--------|
| 全用Opus 4.6 | ~¥8,000-15,000 | ~¥96,000-180,000 |
| 全用Sonnet 4.6 | ~¥2,000-5,000 | ~¥24,000-60,000 |
| 全用Kimi K2.5 | ~¥300-800 | ~¥3,600-9,600 |
| **混合方案（推荐）** | **~¥800-2,000** | **~¥9,600-24,000** |

> 混合方案比全用Sonnet省60-70%成本，比全用Opus省90%+，同时保证精准任务的输出质量。

#### 为什么不全用最便宜的Kimi

| 任务类型 | 全用Kimi的风险 | 混合方案的处理 |
|---------|--------------|--------------|
| 成本计算解读 | 可能对数字推理不够精确，误导代理商 | Sonnet精确解读，数据可信 |
| 知识库问答 | 可能"补充"知识库中不存在的信息 | Sonnet更严格遵循"知识库优先"指令 |
| 客户方案 | 结构可能不够严谨，引用数据可能不准 | Sonnet结构化输出更可控 |
| 话术/内容 | ✅ 完全没问题，中文创作能力强 | Kimi发挥所长 |

## 九、竞争差异性

### 为什么这个方案有创新性

> **核心创新不在于"用AI生成文案"（任何公司都能做），而在于三层递进的结构性变革：**

| 层级 | 创新点 | 竞品能复制吗 |
|------|--------|------------|
| **第一层：体验创新** | "一键备战"统一入口，一次输入全套输出 | 能复制UI，但需要深度的业务理解才能做好编排 |
| **第二层：成本结构创新** | AI让渠道赋能边际成本趋零，服务竞品无法覆盖的市场 | 大平台的利润结构决定了它们不会这样做 |
| **第三层：数据壁垒创新** | 学习飞轮积累"什么话术在什么场景转化率最高"的实战数据 | 没有Ksher的代理商生态，就没有这个数据 |

### vs 直接用ChatGPT/Claude

| 维度 | 通用AI（ChatGPT/Claude） | AgentAI |
|------|------------------------|---------|
| 知识 | 通用知识，不懂Ksher产品细节 | 注入40+文档的Ksher专属知识库 |
| 输出 | 每次只做一件事 | 一次输出全套作战包（话术+成本+方案+异议） |
| 双战场 | 不理解银行客户vs竞品客户的区别 | 自动判断战场，调整全套策略 |
| 数据 | 没有转化率数据 | 积累实战数据，越用越精准 |

## E3. 多Agent协同设计

### E3.1 为什么需要多Agent而不是单次API调用

| 方案 | 优点 | 缺点 |
|------|------|------|
| 单次调用（一个大Prompt） | 简单 | Token爆炸（40+文档全注入=超限）、输出质量下降、无法并行 |
| **多Agent协同** | 每个Agent专注一个任务、按需注入知识库、可并行加速、输出质量高 | 编排复杂度略高 |

### E3.2 Agent定义

系统共7个专业Agent + 1个编排Agent，按任务类型分配不同模型：

| Agent | 职责 | 模型 | 注入的知识库 | 温度 | 预估Token |
|-------|------|------|------------|------|----------|
| **Orchestrator（编排Agent）** | 解析输入→判断战场→分发任务→组装输出 | 无（规则驱动） | 无（Python规则引擎） | — | — |
| **SpeechAgent（话术Agent）** | 生成30秒/3分钟/微信跟进话术 | **Kimi K2.5** | 公司介绍 + 对应业务文档 + 痛点全景图 | 0.7 | ~2K |
| **CostAgent（成本Agent）** | 5项总成本计算 + 可视化数据 | **Sonnet 4.6** | 费率参数JSON + 竞品费率数据 | 0.3 | ~1.5K |
| **ProposalAgent（方案Agent）** | 生成完整客户提案 | **Sonnet 4.6** | 公司介绍 + 对应业务+国家文档 + 特色产品 | 0.5 | ~3K |
| **ObjectionAgent（异议Agent）** | 预判Top3异议 + 回复策略 | **Kimi K2.5** | 竞品对比表 + 异议处理手册 + 优势策略 | 0.7 | ~1.5K |
| **ContentAgent（内容Agent）** | 朋友圈/小红书/视频脚本 | **Kimi K2.5** | 公司介绍 + 痛点全景图 + 行业案例 | 0.8 | ~2K |
| **KnowledgeAgent（知识Agent）** | 产品知识问答 | **Sonnet 4.6** | **全量知识库**（按问题动态检索最相关文档） | 0.3 | ~2K |
| **DesignAgent（设计Agent）** | 海报文案+PPT结构生成 | **Kimi K2.5** | 公司介绍 + 品牌规范 | 0.6 | ~1.5K |

> **模型分配原则**：创意型任务（话术/内容/设计/异议）→ Kimi K2.5（便宜+中文强）；精准型任务（成本计算/方案/知识问答）→ Sonnet 4.6（推理准确+不编造）。两类API通过`LLMClient`统一封装，Agent无需关心底层调用哪个模型。

### E3.2.1 多模型统一客户端（技术实现）

原`services/claude_client.py` 改为 `services/llm_client.py`，封装多模型路由：

```python
# services/llm_client.py — 统一多模型客户端

import os
import anthropic
from openai import OpenAI

# 模型配置
MODEL_CONFIG = {
    "kimi": {
        "client_type": "openai_compatible",
        "base_url": "https://api.moonshot.cn/v1",
        "api_key": os.getenv("KIMI_API_KEY"),
        "model": "kimi-k2.5",
    },
    "sonnet": {
        "client_type": "anthropic",
        "api_key": os.getenv("ANTHROPIC_API_KEY"),
        "model": "claude-sonnet-4-6-20250514",
    },
}

# Agent → 模型映射（核心路由表）
AGENT_MODEL_MAP = {
    "speech":    "kimi",      # 话术 → 创意型
    "content":   "kimi",      # 内容 → 创意型
    "design":    "kimi",      # 设计 → 创意型
    "objection": "kimi",      # 异议 → 创意型
    "cost":      "sonnet",    # 成本 → 精准型
    "proposal":  "sonnet",    # 方案 → 精准型
    "knowledge": "sonnet",    # 知识 → 精准型
}

class LLMClient:
    """统一接口，底层自动路由到不同模型"""

    def __init__(self):
        self._clients = {}

    def _get_client(self, model_key: str):
        if model_key not in self._clients:
            config = MODEL_CONFIG[model_key]
            if config["client_type"] == "openai_compatible":
                self._clients[model_key] = OpenAI(
                    base_url=config["base_url"],
                    api_key=config["api_key"]
                )
            else:
                self._clients[model_key] = anthropic.Anthropic(
                    api_key=config["api_key"]
                )
        return self._clients[model_key]

    def call(self, agent_name: str, system: str, user_msg: str,
             temperature: float = 0.7, stream: bool = True):
        """
        统一调用接口 — Agent只需传agent_name，自动路由到对应模型
        """
        model_key = AGENT_MODEL_MAP[agent_name]
        config = MODEL_CONFIG[model_key]
        client = self._get_client(model_key)

        if config["client_type"] == "openai_compatible":
            return client.chat.completions.create(
                model=config["model"],
                messages=[
                    {"role": "system", "content": system},
                    {"role": "user", "content": user_msg}
                ],
                temperature=temperature,
                stream=stream
            )
        else:
            return client.messages.create(
                model=config["model"],
                max_tokens=4096,
                system=system,
                messages=[{"role": "user", "content": user_msg}],
                temperature=temperature,
                stream=stream
            )
```

**⚠️ 关键：统一流式输出接口**

OpenAI SDK和Anthropic SDK的流式返回格式完全不同。`LLMClient`必须提供统一的流式文本接口，让调用方无需关心底层SDK：

```python
# services/llm_client.py — 统一流式输出（追加到LLMClient类中）

def stream_text(self, agent_name: str, system: str, user_msg: str,
                temperature: float = 0.7):
    """统一流式输出接口：yield纯文本chunk，调用方无需关心底层SDK差异"""
    model_key = AGENT_MODEL_MAP[agent_name]
    config = MODEL_CONFIG[model_key]
    client = self._get_client(model_key)

    if config["client_type"] == "openai_compatible":
        response = client.chat.completions.create(
            model=config["model"],
            messages=[
                {"role": "system", "content": system},
                {"role": "user", "content": user_msg}
            ],
            temperature=temperature,
            stream=True
        )
        for chunk in response:
            if chunk.choices and chunk.choices[0].delta.content:
                yield chunk.choices[0].delta.content
    else:  # anthropic
        with client.messages.stream(
            model=config["model"],
            max_tokens=4096,
            system=system,
            messages=[{"role": "user", "content": user_msg}],
            temperature=temperature,
        ) as stream:
            for text in stream.text_stream:
                yield text

def call_sync(self, agent_name: str, system: str, user_msg: str,
              temperature: float = 0.7) -> str:
    """同步调用（非流式），返回完整文本。用于CostAgent等需要完整结果后再传递给下游Agent的场景"""
    chunks = []
    for chunk in self.stream_text(agent_name, system, user_msg, temperature):
        chunks.append(chunk)
    return "".join(chunks)
```

**Agent调用方式**（所有Agent统一，无需关心底层模型）：

```python
# agents/base_agent.py — Agent抽象基类
from abc import ABC, abstractmethod

class BaseAgent(ABC):
    def __init__(self, llm_client, knowledge_loader, agent_name: str):
        self.llm_client = llm_client
        self.knowledge_loader = knowledge_loader
        self.agent_name = agent_name

    @abstractmethod
    def generate(self, context: dict) -> dict:
        """同步生成，返回结构化结果dict"""
        pass

    def stream(self, context: dict):
        """流式生成，yield文本chunk（UI直接消费）"""
        knowledge = self.knowledge_loader.load(self.agent_name, context)
        system = self.build_system_prompt(knowledge)
        user_msg = self.build_user_message(context)
        yield from self.llm_client.stream_text(
            self.agent_name, system, user_msg, self.temperature
        )

    @abstractmethod
    def build_system_prompt(self, knowledge: str) -> str:
        pass

    @abstractmethod
    def build_user_message(self, context: dict) -> str:
        pass
```

### E3.4.1 Agent间数据流转与半并行执行

```
Orchestrator 的战场判断
    │
```

> **⚠️ 自检发现的技术盲点**：ProposalAgent依赖CostAgent的输出，但4个Agent不能完全并行。实际执行是**两阶段半并行**。

**半并行执行实现（Orchestrator核心逻辑）**：

```python
# orchestrator/agent_dispatcher.py

from concurrent.futures import ThreadPoolExecutor, as_completed

def generate_battle_pack(context: dict, agents: dict) -> dict:
    """
    两阶段半并行：
    阶段1：Speech + Cost + Objection 并行（3个Agent互不依赖）
    阶段2：Proposal 串行（依赖Cost的输出）
    """
    results = {}

    # 阶段1：3个Agent并行
    with ThreadPoolExecutor(max_workers=3) as executor:
        futures = {
            executor.submit(agents["speech"].generate, context): "speech",
            executor.submit(agents["cost"].generate, context): "cost",
            executor.submit(agents["objection"].generate, context): "objection",
        }
        for future in as_completed(futures):
            agent_name = futures[future]
            try:
                results[agent_name] = future.result(timeout=60)
            except Exception as e:
                results[agent_name] = {"error": str(e)}

    # 阶段2：Cost完成后，注入结果给Proposal
    if "error" not in results.get("cost", {}):
        context["cost_data"] = results["cost"]
        context["annual_saving"] = results["cost"].get("annual_saving", 0)

    try:
        results["proposal"] = agents["proposal"].generate(context)
    except Exception as e:
        results["proposal"] = {"error": str(e)}

    return results
```

> **为什么不用asyncio**：Streamlit基于自己的事件循环运行，`asyncio.run()`会报"事件循环已运行"错误。`ThreadPoolExecutor`是Streamlit兼容的并行方案。

### E3.5 三层知识融合机制（所有Agent通用）

> **核心问题**：CLI终端中Claude可以自己读文件、联网搜索，但API调用中Claude只能基于你注入的内容+自身训练知识来回答。因此必须通过Prompt设计实现"知识库精准数据 + 模型自身行业知识"的智能融合，让输出质量超过单独使用任何一层知识。

#### 为什么需要融合而不是只用知识库

```
代理商实际问题的三种类型：

类型A：知识库能直接回答
  "泰国B2C费率多少？" → 知识库有精确数据 → 直接引用

类型B：知识库没有，但Claude自身知识能回答
  "2025年东南亚跨境电商增长了多少？" → 行业公开数据 → 模型知识补充

类型C：需要两者结合（最常见、最有价值）
  "为什么应该选Ksher做泰国市场？"
  → 知识库：SCB本地账户+秒到宝+0.8%费率
  → 模型知识：泰国是东南亚第二大电商市场+Shopee泰国GMV增速
  → 融合输出：行业背景+Ksher差异化优势+可操作建议
```

**如果只用知识库**：回答像"查手册"，缺乏行业深度，说服力弱
**如果只用模型知识**：不知道Ksher产品细节，回答泛泛
**融合后**：既有精准的产品数据弹药，又有行业背景的说服力 — 这才是"AI专家顾问"级别的输出

### E3.6.1 Agent调用失败降级策略

> **自检补充**：4个Agent并行，如果其中1个API调用超时/报错，整个作战包不应该全部失败。

```python
# orchestrator/result_assembler.py

def assemble_battle_pack(results: dict) -> dict:
    """组装作战包，处理部分Agent失败的情况"""
    pack = {"status": "complete", "agents": {}}

    for agent_name in ["speech", "cost", "proposal", "objection"]:
        result = results.get(agent_name, {})
        if "error" in result:
            pack["agents"][agent_name] = {
                "status": "failed",
                "error_msg": "该模块暂时不可用，请稍后重试",
                "retry_available": True
            }
            pack["status"] = "partial"  # 标记为部分完成
        else:
            pack["agents"][agent_name] = {
                "status": "success",
                "data": result
            }

    return pack
```

**UI层处理**：成功的模块正常展示，失败的模块显示友好提示 + 「重试」按钮。代理商看到3/4成功时仍可使用，不会因1个模块卡住整个体验。

### E4.4 知识库动态管理（技术实现）

> 产品迭代时（费率调整/新增国家/流程变更），渠道负责人需要能直接更新知识库，AI回答随之同步变化。

#### V1 实现方案（Streamlit内置）

**原理**：知识库是本地Markdown文件 → `knowledge_loader.py`每次被Agent调用时重新读取文件 → 文件被编辑保存后，下次AI调用自动使用新内容，无需重启。

**新增文件**：

```
ui/pages/
  └── knowledge_admin.py        # 知识库管理页面（管理员入口）
```

**核心实现逻辑**：

```python
# ui/pages/knowledge_admin.py

import streamlit as st
import os
import json
from pathlib import Path

KNOWLEDGE_DIR = "knowledge/"

def render_knowledge_admin():
    st.header("🔧 知识库管理")

    # --- 文档管理 ---
    st.subheader("📁 知识库文档")

    # 列出所有知识库文件
    files = list_knowledge_files(KNOWLEDGE_DIR)
    selected_file = st.selectbox("选择文档", files)

    if selected_file:
        content = Path(KNOWLEDGE_DIR + selected_file).read_text()

        # 编辑区
        edited = st.text_area(
            "编辑内容",
            value=content,
            height=400,
            help="修改后点击保存，AI下次回答将使用新内容"
        )

        col1, col2 = st.columns(2)
        with col1:
            if st.button("💾 保存"):
                Path(KNOWLEDGE_DIR + selected_file).write_text(edited)
                st.success(f"✅ 已保存。AI将使用更新后的内容。")
        with col2:
            if st.button("👁️ 预览AI效果"):
                # 用更新后的知识测试一个问题
                test_ai_with_updated_knowledge(edited)

    # --- 费率参数管理（结构化表单）---
    st.subheader("💰 费率参数")

    fee_data = json.loads(Path(KNOWLEDGE_DIR + "fee_structure.json").read_text())

    for provider, rates in fee_data.items():
        with st.expander(f"{provider} 费率"):
            for key, value in rates.items():
                fee_data[provider][key] = st.number_input(
                    f"{key}", value=value, key=f"{provider}_{key}"
                )

    if st.button("💾 保存费率"):
        Path(KNOWLEDGE_DIR + "fee_structure.json").write_text(
            json.dumps(fee_data, indent=2, ensure_ascii=False)
        )
        st.success("✅ 费率已更新。成本计算器将使用新费率。")

    # --- 上传新文档 ---
    st.subheader("📤 上传新文档")
    uploaded = st.file_uploader("上传Markdown文件", type=["md"])
    target_dir = st.selectbox("目标目录",
        ["base/", "b2c/", "b2b/", "service_trade/",
         "products/", "competitors/", "operations/", "strategy/"])

    if uploaded and st.button("上传"):
        save_path = KNOWLEDGE_DIR + target_dir + uploaded.name
        Path(save_path).write_text(uploaded.read().decode())
        # 同步更新index.json
        update_index(save_path, target_dir)
        st.success(f"✅ 已上传到 {save_path}")
```

**关键设计点**：

| 设计点 | 实现方式 | 为什么这样做 |
|--------|---------|------------|
| 改完即生效 | `knowledge_loader`每次调用都读文件，不做缓存 | 最简单的"热更新"——不需要重启、不需要部署 |
| 费率单独入口 | JSON结构化表单（改数字即可） | 费率更新最频繁、影响最大，不能让用户去改JSON原文 |
| 预览AI效果 | 保存后可输入测试问题验证 | 改完立即验证，避免"改了但不确定AI用没用新内容" |
| 上传新文档 | 文件上传 + 选目录 + 自动更新index.json | 新增国家/竞品时不需要碰代码 |

## E7. Agent Teams 多终端协同开发

> **核心问题**：这个项目涉及架构设计、后端开发、前端UI、知识库工程、Prompt设计、测试等多个专业领域，单个Agent串行工作太慢（7天）。采用Agent Teams多终端并行，**一台MacBook开多个终端窗口**，不同角色的AI Agent各司其职、同步推进。

### E7.1 为什么需要Agent Teams

| 方案 | 耗时 | 问题 |
|------|------|------|
| 单Agent串行 | 7天（每天6-8h） | 一个Agent写后端时，前端/知识库/Prompt都在等 |
| **多Agent并行** | **4-5天** | 后端写代码的同时，另一个Agent在整理知识库，第三个在写Prompt |

## E11. 执行纪律：「阻塞即停」机制

> **核心原则**：执行过程中遇到任何卡点，**立即停下来**，明确卡点和所需条件，等条件准备好后再继续。绝不跳过障碍强行推进，绝不用占位数据凑合。

### 为什么需要这个机制

```
错误的执行方式：
  遇到问题 → 假设条件满足 → 用占位数据继续 → 后面全部基于错误假设 → 返工

正确的执行方式：
  遇到问题 → 停下来 → 明确需要什么 → 准备好 → 继续 → 每一步都扎实
```

Demo的质量取决于每一步的扎实程度。跳过一个前置条件，后面所有依赖它的模块都会出问题。**宁可慢一天做对，不要快一天返工。**

### 阻塞检查点清单

以下是开发过程中可能触发「阻塞即停」的关键检查点：

#### Day 0 阻塞点

|--------|---------|---------|---------|

| 知识库源文件 | Obsidian Vault中37个核心文档无法访问 | 确认文件路径和访问权限 | **停**，无知识库则AI输出无意义 |

### 阻塞处理流程

```
开发中遇到问题
       │
       ▼
  判断：是否影响核心功能？
       │
  ┌────┴────┐
  │ 是      │ 否
  │         │
  ▼         ▼
立即停止    记录为TODO
  │         继续开发
  ▼
输出「阻塞报告」：
  ① 卡在哪一步
  ② 具体什么问题
  ③ 需要准备什么条件
  ④ 预计准备好后恢复时间
  ⑤ 是否有降级方案
       │
       ▼
等待条件准备好
       │
       ▼
验证条件 → 继续执行
```

### 可降级 vs 必须停的判断标准

| 类型 | 判断标准 | 处理方式 |
|------|---------|---------|
| **必须停** | 影响核心演示流程（一键备战、双战场、知识问答） | 停下来解决，不继续 |
| **必须停** | 数据准确性问题（费率/成本计算） | 宁可延期，不能用错误数据 |
| **可降级** | 锦上添花功能（PPT生成、海报设计） | 标记"V2实现"，Demo中简化处理 |
| **可降级** | UI美化问题 | 功能正确优先，样式后调 |

> **执行口号：宁可停下来做对，不要赶进度做错。Demo的可信度 > Demo的完整度。**

## E12. 关键决策记录

| # | 决策 | 选择 | 理由 | 替代方案 |
|---|------|------|------|---------|
| D1 | Prompt注入 vs RAG | Prompt注入 | Demo阶段知识量可控（<100K tokens），RAG增加复杂度 | V2用RAG |
| D2 | 多Agent并行 vs 串行 | 并行（CostAgent先返回后注入Proposal） | 减少等待时间，提升Demo体验 | 全串行更简单但慢 |
| D3 | Orchestrator用LLM vs 规则 | 规则引擎 | 战场判断逻辑简单确定，LLM不必要 | V2可升级为LLM路由 |
| D4 | 反馈数据用JSON vs DB | JSON | Demo阶段足够，避免引入数据库依赖 | V2用PostgreSQL |
| D5 | Sonnet vs Opus | Sonnet 4.6 | 速度快+成本低，Demo质量足够 | 关键模块可选Opus |
| D6 | 一键备战 vs 模块并列 | 一键备战为核心 | 乔布斯启示：统一体验>功能菜单 | — |
| D7 | Kimi模型选择 | kimi-k2.5（非kimi-2.6） | 2.6是App版本号不是API模型名，k2.5是最新最强版本 | kimi-k2-turbo-preview（速度优先） |
| D8 | 流式输出统一 | LLMClient.stream_text()封装 | OpenAI/Anthropic流式协议完全不同，必须在服务层统一 | 让每个Agent自己处理（重复代码） |
