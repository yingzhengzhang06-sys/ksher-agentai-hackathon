# 开发安全边界与执行纪律

## 1. 禁止事项

- 不允许 `git add .`。
- 不允许自动提交代码。
- 不允许修改 `.env`、密钥、Token、生产配置。
- 不允许直接部署生产。
- 不允许扩大 MVP 范围。
- 不允许无验收标准开发。
- 不允许无测试用例上线。
- 不允许读取或输出 API Key、Token、SSH Key。
- 不允许把未提交历史改动混入当前任务提交。
- 不允许在未授权情况下修改任务单外文件。

## 2. 文件修改原则

- 每个任务必须声明允许修改文件。
- 每个任务必须声明禁止修改文件。
- 修改范围不得超出任务单。
- 如发现必须超范围，停止执行并标注阻塞。
- 不确定文件归属时，先读取任务单和协作状态文档。
- 遇到已有未提交改动时，不得回滚，不得覆盖，必须与当前任务隔离处理。

任务执行前必须确认：

- 当前任务编号。
- 当前责任角色。
- 输入文档。
- 允许修改文件。
- 禁止修改文件。
- 测试命令。
- 验收标准。

## 3. 测试原则

- 每个开发任务必须运行最小测试。
- 关键阶段必须运行全量回归。
- 测试失败不得进入下一阶段。
- warnings 可以记录，但不能掩盖失败。
- 测试无法运行时必须说明原因、影响范围和建议处理人。
- QA 未完成验收时，不得宣称功能最终通过。

## 4. MVP 范围控制

以当前“发朋友圈数字员工 / 朋友圈转发助手”功能为例，本期明确不允许：

- 真实微信发布接口。
- 自动发布。
- 定时发布。
- 审批流。
- CRM。
- 素材库。
- 数据分析闭环。
- 多账号系统。
- 大规模重构。
- 海报生成。
- 配图生成。
- 短视频脚本。
- 渠道商门户。

如后续需要新增上述能力，必须新建版本规划、PRD、Tech Design、Tasks 和 Test Cases，不得混入当前 MVP。
