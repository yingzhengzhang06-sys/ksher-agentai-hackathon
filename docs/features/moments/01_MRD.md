# 发朋友圈数字员工 MRD

> 文档版本：v1.0  
> 更新日期：2026-04-24  
> 功能名称：发朋友圈数字员工  
> 所属项目：Ksher AgentAI 智能工作台 / 市场专员数字员工 / 内容工厂  
> 当前阶段：MVP 定义与交付口径  
> 目标读者：产品负责人、产品经理、设计师、AI 能力设计师、工程师、测试工程师

## 1. 产品背景

跨境支付渠道商、区域 BD、市场专员日常需要通过微信朋友圈、社群、私聊等私域渠道触达商户。内容发布频率高，但渠道商通常不具备持续产出专业金融科技内容的能力，总部市场团队也难以为每个渠道、每类客户实时定制话术。

“发朋友圈数字员工”选择一个高频、低门槛、可验证的动作作为 MVP：用户输入内容类型、目标客户、产品卖点和文案风格，系统生成一条可复制到朋友圈的专业、克制、合规文案草稿，并给出转发建议和合规提示。

本功能不是完整营销自动化平台，而是先打通“输入业务意图 -> AI 生成私域内容 -> 基础合规检查 -> 人工确认复制发布 -> 反馈优化”的闭环。

## 2. 一句话定位

帮助跨境支付渠道商在 3 分钟内生成一条可以复制到朋友圈的专业、自然、低合规风险的营销内容草稿。

## 3. 产品目标

| 目标 | 描述 | MVP 验收口径 |
|---|---|---|
| 降低内容生产门槛 | 渠道商无需从零写跨境支付朋友圈 | 用户填写 4 类核心输入即可生成内容 |
| 提升私域表达质量 | 内容不像硬广，能体现目标客户场景 | 正文包含目标客户和至少 1 个卖点 |
| 控制合规风险 | 避免绝对化、收益承诺、规避监管等表达 | 输出包含合规提示和改写建议 |
| 支持快速复制使用 | 生成结果可直接复制到微信朋友圈 | 提供“复制文案”“重新生成”操作 |
| 建立反馈闭环 | 收集有用/没用反馈用于后续优化 | 记录生成 ID、反馈类型、负反馈原因 |

## 4. 目标用户

### 4.1 第一优先级用户

| 用户 | 特征 | 核心诉求 |
|---|---|---|
| 城市代理商 / 区域渠道商 | 有本地商户资源，日常通过微信维护客户，但不擅长内容生产 | 每天有可复制的专业内容可发 |
| 跨境服务商渠道伙伴 | ERP、物流、财税、培训、独立站服务商，想顺手推广支付服务 | 不研究金融话术也能安全表达产品价值 |

### 4.2 第二优先级用户

| 用户 | 特征 | 核心诉求 |
|---|---|---|
| 总部市场专员 | 需要批量支持渠道商拿素材、改文案、发内容 | 降低重复文案支持成本 |
| 渠道负责人 / 销售支持 | 需要提升渠道活跃度和内容质量 | 让渠道商自助拿内容，提升执行稳定性 |

## 5. 用户痛点

### 5.1 渠道商不会持续产出内容

常见替代方式是转发总部公众号、复制竞品文案、让销售临时写、用通用 AI 生成。这些方式通常不够私域化、不够贴近客户、表达泛泛，且缺少稳定发布节奏。

### 5.2 跨境支付表达有专业门槛

跨境支付内容涉及多币种、到账时效、费率、清算路径、支付牌照、合规要求、平台规则等概念。渠道商知道产品有价值，但很难用商户听得懂、又不过度承诺的语言讲清楚。

### 5.3 金融和支付内容有合规风险

朋友圈文案不能随意使用“最低费率”“最快到账”“最安全”“保证收益”“零风险”等绝对化表达，也不能暗示规避监管、夸大到账时效、未经授权引用客户或竞品。

### 5.4 总部素材供给跟不上私域节奏

总部内容通常偏官方、偏品牌传播，不能覆盖渠道商每天面对的具体客户类型和销售场景。渠道商需要的是短、快、可复制、能像自己说出来的内容。

## 6. 核心场景

| 场景 | 用户意图 | 系统响应 |
|---|---|---|
| 日常朋友圈更新 | 今天想发一条专业内容保持曝光 | 生成自然、轻量 CTA 的朋友圈正文 |
| 产品卖点推广 | 想围绕到账快、费率透明、合规安全发一条内容 | 根据卖点生成场景化表达 |
| 目标客户定制 | 面向 Amazon、Shopee、外贸 B2B 不同客户群发内容 | 调整痛点、措辞、例子和转发建议 |
| 总部批量支持 | 市场专员要给渠道商提供可用素材 | 快速生成多版可复制草稿并收集反馈 |

## 7. MVP 范围

### 7.1 本期包含

| 能力 | 说明 |
|---|---|
| 输入表单 | 内容类型、目标客户、产品卖点、文案风格、补充说明 |
| 文案生成 | 输出标题/首句、正文、转发建议、合规提示、改写建议 |
| 结构化输出 | 后端返回固定 JSON 结构，便于前端展示和测试 |
| 输入校验 | 必填项、产品卖点数量、补充说明 300 字限制 |
| 基础质量检查 | 完整性、长度、敏感表达、目标客户和卖点覆盖 |
| 失败兜底 | AI 失败、空输出、格式异常时返回“需要人工补充”模板 |
| 操作按钮 | 生成、复制文案、重新生成、有用、没用、提交反馈 |
| 反馈记录 | 支持 positive / negative 反馈和负反馈原因 |

### 7.2 本期不包含

| 不包含能力 | 原因 |
|---|---|
| 自动发布到微信朋友圈 | 微信接口限制和合规风险高，MVP 只做人工复制 |
| 定时发布、多账号管理 | 属于营销自动化平台范畴，超出样板功能 |
| 配图、海报、短视频生成 | 另属内容工厂其他模块 |
| CRM 线索跟进和转化归因 | 需要数据链路和客户权限设计 |
| 审批流和组织权限 | 先验证单人生成体验 |
| 外部热点实时抓取 | MVP 优先稳定输入生成，热点可后续扩展 |

## 8. 成功指标

| 指标 | MVP 目标 |
|---|---|
| 生成完成时间 | 常规请求 20 秒内完成，Mock 联调 2 秒内返回 |
| 可复制率 | 生成结果中正文可独立复制，不依赖辅助说明 |
| 字段完整率 | 成功响应必须包含 5 类输出字段 |
| 安全拦截率 | 明显绝对化或承诺性表达被标记为需改写 |
| 反馈入口可用性 | 成功生成后可提交有用/没用反馈 |
| 测试覆盖 | 模型、Prompt、服务解析、API 错误状态均有测试 |

## 9. 核心假设

| 假设 | 验证方式 |
|---|---|
| 渠道商愿意用结构化表单换取更稳定结果 | 观察生成点击率和重新生成次数 |
| 80-220 字正文更适合朋友圈复制 | 收集有用率和负反馈原因 |
| 合规提示不会显著打断使用体验 | 观察复制前是否忽略高风险提示 |
| Mock -> 真实 LLM 的接口结构可保持稳定 | 使用 Pydantic 模型和 Prompt 单测锁定结构 |

## 10. 风险与约束

| 风险 | 影响 | 对策 |
|---|---|---|
| AI 输出夸大产品能力 | 可能造成合规和品牌风险 | Prompt 禁止、质量检查、需改写状态 |
| 输出太像广告 | 用户不愿复制发布 | 风格参数和负反馈原因优化 |
| 输入过少导致内容空泛 | 生成质量下降 | 必填结构化字段，补充说明可选 |
| 真实 LLM 不稳定 | 影响生成体验 | 统一 LLM 状态、熔断、Mock/兜底模板 |
| 用户误以为系统自动审核通过 | 合规责任误解 | 明确“草稿”和“人工确认后发布”边界 |

## 11. 后续迭代方向

| 阶段 | 方向 |
|---|---|
| V1.1 | 增加更多目标客户、卖点、风格模板和行业案例 |
