# 数字员工项目协作协议

## 1. 目标

本协议用于让产品负责人、架构师、工程师、QA 测试工程师、项目助理 Bot 基于同一事实源协作，减少人工反复转述，降低跨角色交接中的信息丢失和范围失控风险。

适用范围：

- 数字员工项目的功能规划、设计、开发、测试、验收和复盘
- 单功能从 MRD 到 QA 验收的完整协作链路
- ChatGPT / Codex / Claude Code / OpenClaw / Telegram Bot 等 AI 协作工具的角色边界约束

## 2. 共同事实源

项目协作必须以文档和状态文件作为事实源，不以聊天窗口记忆作为最终判断依据。

- `docs/` 是项目事实源。
- `docs/features/<feature>/` 是单功能事实源。
- `docs/features/<feature>/08_Collaboration_Status.md` 是当前功能协作状态事实源。
- `prompts/roles/` 是角色提示词事实源。
- 每个阶段的最终结论必须写入固定路径的 Markdown 文件。
- 聊天窗口可以用于讨论、澄清和临时指令，但不能替代文档事实源。
- AI 可以参与生产，但不能成为未落文档的事实源。

当聊天内容与文档事实源冲突时：

1. 优先读取文档事实源。
2. 在输出中标注冲突点。
3. 交由产品负责人或架构师裁决。
4. 不允许自行扩大 MVP 范围。

## 3. 五个核心角色

本项目默认只保留五个核心角色，不再为 MVP 阶段扩展成更多常驻角色，避免流程过重。发布、合规、环境、UIUX 等职责作为“职责帽子”挂靠到现有角色中，只有当后续任务复杂度明显上升时，才由产品负责人和架构师共同决定是否拆成独立角色。

### 3.1 产品负责人

- 主要职责：
  - 控制产品方向和 MVP 范围。
  - 做阶段门禁评审。
  - 判断是否允许进入下一阶段。
  - 判断是否允许上线、演示或归档。
- 输入文档：
  - MRD
  - PRD
  - UIUX
  - AI Design
  - Tech Design
  - Test Cases
  - Collaboration Status
- 输出文档：
  - 产品门禁评审
  - 范围控制意见
  - 下一阶段准入结论
- 不允许做的事：
  - 不直接写代码。
  - 不绕过 QA。
  - 不用聊天记忆替代文档事实源。
  - 不因个人偏好扩大 MVP。
- 交接对象：
  - 架构师 / 技术负责人
  - QA 测试工程师
  - 项目助理 Bot

### 3.2 架构师 / 技术负责人

- 主要职责：
  - 设计技术方案。
  - 拆分工程任务。
  - 审核工程实现。
  - 做技术裁决。
  - 控制技术风险和 MVP 边界。
- 输入文档：
  - PRD
  - UIUX
  - AI Design
  - Tech Design
  - Engineering Subtasks
  - Test Cases
  - Collaboration Status
- 输出文档：
  - 技术方案
  - 工程任务拆解
  - 技术审核结论
  - 技术裁决记录
  - 下一步工程指令
- 不允许做的事：
  - 不直接扩大业务范围。
  - 不让工程师无边界改代码。
  - 不允许绕过测试。
  - 不修改密钥或生产配置。
- 交接对象：
  - 工程师
  - QA 测试工程师
  - 产品负责人

### 3.3 工程师

- 主要职责：
  - 按任务单实现代码或文档。
  - 严格遵守允许修改范围。
  - 执行最小测试和必要回归测试。
  - 输出完成报告。
  - 遇到阻塞立即停止并反馈。
- 输入文档：
  - Engineering Subtasks
  - Tech Design
  - Test Cases
  - Collaboration Status
  - 具体任务单
- 输出文档：
  - 修改文件列表
  - 实现内容摘要
  - 测试命令和结果
  - 风险 / 阻塞
  - 下一建议角色和任务
- 不允许做的事：
  - 不使用 `git add .`。
  - 不自动提交代码。
  - 不修改 `.env`、密钥、Token、生产配置。
  - 不扩大 MVP。
  - 不修改任务单外文件。
  - 不跳过测试。
- 交接对象：
  - 架构师 / 技术负责人
  - QA 测试工程师
  - 项目助理 Bot

### 3.4 QA 测试工程师

- 主要职责：
  - 执行测试用例。
  - 执行人工验收。
  - 记录缺陷。
  - 复验缺陷修复。
  - 判断是否允许进入产品负责人门禁。
- 输入文档：
  - Test Cases
  - Collaboration Status
  - Engineering Subtasks
  - 当前构建 / 当前页面 / 当前 API
- 输出文档：
  - 测试结果
  - 缺陷记录
  - 截图路径
  - 复现步骤
  - 责任角色
  - 是否允许进入产品负责人门禁
- 不允许做的事：
  - 不主动改业务代码。
  - 不修改 `.env`、密钥、Token、生产配置。
  - 不跳过失败用例。
  - 不用“看起来可以”代替测试记录。
- 交接对象：
  - 工程师
  - 架构师 / 技术负责人
  - 产品负责人

### 3.5 项目助理 Bot

- 主要职责：
  - 维护任务状态。
  - 汇总当前进展。
  - 判断下一角色。
  - 提醒阻塞项。
  - 确保文档事实源同步。
- 输入文档：
  - Collaboration Status
  - Engineering Subtasks
  - Test Cases
  - 最新完成报告
- 输出文档：
  - 当前阶段
  - 当前负责人
  - 下一负责人
  - 下一任务
  - 阻塞项
  - 需要同步更新的文件
- 不允许做的事：
  - 不改业务代码。
  - 不做产品决策。
  - 不做技术裁决。
  - 不跳过产品负责人或 QA 门禁。
- 交接对象：
  - 产品负责人
  - 架构师 / 技术负责人
  - 工程师
  - QA 测试工程师

### 3.6 职责帽子挂靠规则

以下职责不作为 MVP 阶段的新增核心角色，而是挂靠到现有五个角色：

| 职责帽子 | 默认挂靠角色 | 主要职责 | 不新增流程节点的原因 |
|---|---|---|---|
| 发布负责人 / Release Owner | 产品负责人 + 项目助理 Bot | 判断是否可演示、是否可发 PR、发布记录是否完整 | 当前 MVP 只需要门禁和记录，不需要独立发布团队 |
| 安全与合规审核人 | 架构师 / 技术负责人 | 检查不接真实微信、不碰密钥、不输出高风险金融承诺、不扩大 MVP | 当前合规边界已经体现在技术方案和 QA 用例中 |
| 运维 / 环境负责人 | 工程师 + 项目助理 Bot | 执行开发前检查、依赖检查、测试基线和脚本状态检查 | 当前仅需本地开发环境门禁，不涉及生产运维 |
| UIUX 设计师 | 产品负责人 + 工程师 | 维护低保真原型、交互说明和前端验收映射 | 当前前端已进入工程实现阶段，不作为常驻流程节点 |

职责帽子执行规则：

- 不单独增加任务流转状态。
- 不绕过五个核心角色的交接和门禁。
- 不允许以职责帽子名义扩大 MVP。
- 如某职责需要独立产出文档，仍写入 `docs/` 或 `docs/features/<feature>/` 固定路径。
- 如某职责需要独立角色，必须先由产品负责人判断流程收益，再由架构师确认执行边界。

## 4. 标准协作流

主流程：

```text
产品负责人 -> 架构师 -> 工程师 -> QA -> 产品负责人门禁 -> 复盘
```

缺陷流：

```text
QA -> 责任工程师 -> QA 复验 -> 产品负责人门禁
```

阶段说明：

1. 产品负责人确认目标、MVP 范围和验收标准。
2. 架构师设计技术方案、拆分任务并设定修改边界。
3. 工程师按单任务执行，完成后输出交接报告。
4. QA 按测试用例和验收 Runbook 执行测试。
5. 产品负责人根据 QA 结果做最终门禁。
6. 项目完成后输出复盘，沉淀经验和后续优化项。

## 5. 任务交接规则

每个任务完成后必须输出交接报告。

交接报告必须说明：

- 任务编号和任务名称。
- 执行角色。
- 修改文件。
- 未修改范围声明。
- 测试命令。
- 测试结果。
- 风险和阻塞。
- 下一建议角色。
- 下一建议任务。

如果存在阻塞，必须标注：

- 阻塞原因。
- 影响范围。
- 责任角色。
- 建议处理方式。
- 是否允许继续下一阶段。

## 6. 门禁规则

- 没有 MRD / PRD，不进入设计。
- 没有 UIUX / AI Design，不进入技术方案。
- 没有验收标准，不进入开发。
- 没有测试用例，不允许上线。
- QA 未通过，不允许产品负责人最终门禁通过。
- 架构师未明确允许修改范围，不允许工程师开始编码。
- 存在 P0 / P1 阻塞缺陷，不允许进入上线或演示门禁。

## 7. 执行纪律

- 不允许 `git add .`。
- 不允许自动提交代码。
- 不允许修改 `.env`、密钥、Token、生产配置。
- 不允许扩大 MVP 范围。
- 不允许未授权部署。
- 不允许无验收标准开发。
- 不允许无测试用例上线。
- 不允许把未提交历史改动混入当前任务提交。
- 每次任务必须明确允许修改文件和禁止修改文件。
- 每次任务完成后必须运行最小测试；关键阶段必须运行全量回归。
