# Ksher AgentAI 智能工作台 — 朋友圈素材发布系统设计方案

> **核心理念**：AI 生成 → 人工精品。用设计团队的真实物料替代 AI 生成内容，打造"每周 5 条精品朋友圈"的极致体验。
> **当前阶段**：产品设计阶段。仅实现设计人员后台上传工具，暂不涉及渠道商使用，暂不涉及账号体系。
> **改造范围**：仅「7 天日历」模块的后台上传功能。朋友圈其他功能保持不变。
> **集成方式**：后台作为现有平台的一个专属板块，用临时密码保护入口。
> **目标用户（当前）**：设计人员（上传+预览+管理）
> **目标用户（未来）**：渠道商/员工（查看使用）← 暂缓
> **架构定位**：本后台为**通用内容管理中心（Content Admin Center）**的第一个模块。未来扩展：AI 学习材料上传、公司内部文档管理等。
> **状态**：待评审

---

## 1. 现状与痛点

### 1.1 当前 AI 生成模式的不足

| 痛点 | 具体表现 |
|------|---------|
| **AI 味重** | 转发语模板化，"综上所述"、"我们很高兴"等 AI 痕迹明显 |
| **海报不够专业** | HTML 模板渲染 vs 设计师 PS/AI 出品，视觉差距大 |
| **内容不可控** | 每周内容由算法随机生成，无法精准配合营销节奏 |
| **缺乏信任感** | 渠道商觉得内容"太假"，转发意愿低 |
| **无法配合活动** | 公司做促销活动时，AI 无法同步输出对应物料 |

### 1.2 竞品观察

| 平台 | 做法 | 效果 |
|------|------|------|
| PingPong 代理商后台 | 每周推送 5 条固定素材 | 渠道商转发率高，因为内容真实 |
| XTransfer 营销中心 | 人工运营团队每周更新素材库 | 配合产品上新节奏，转化好 |
| 连连国际 | AI + 人工审核混合 | 质量参差不齐 |

**结论**：B2B 跨境支付行业的渠道商更愿意转发"公司官方出品"的物料，而非 AI 生成内容。

---

## 2. 核心设计原则

1. **人工精品优于 AI 批量** — 每周 5 条，每条都是设计团队精心制作
2. **一周一看板** — 按自然周组织，渠道商登录即看到本周全部素材
3. **一键使用** — 复制转发语 + 下载海报原图，两步完成朋友圈发布
4. **数据驱动优化** — 统计查看/下载/复制数据，反哺内容策略
5. **极简后台** — 设计人员 3 分钟完成一条素材上传

## 3. 角色体系更新

在账号体系（auth_system_design.md）基础上，新增 **designer** 角色：

| 角色 | 标识 | 权限 |
|------|------|------|
| **系统管理员** | `admin` | 全部功能 + 素材管理 + 数据报表（当前阶段与 designer 合并处理） |

> **当前阶段说明**：暂不实现完整账号体系。designer 入口用临时密码保护（环境变量配置），密码由管理员单独告知设计人员。未来接入账号体系后，临时密码逻辑无缝替换为正式登录。

## 4. 架构设计：通用内容管理中心

### 4.1 定位与扩展路线

本后台定位为**通用内容管理中心（Content Admin Center）**，当前第一个模块是「朋友圈素材管理」，后续逐步扩展：

```
内容管理中心（admin/designer 专属）
├── 📅 朋友圈素材          ← 当前实现（Phase 1）
│   ├── 每周 5 条素材上传
│   ├── 海报 + 转发语管理
│   └── 素材预览（设计人员自己看效果）
│
├── 📚 AI 学习材料         ← 预留入口（Phase 2，需求待定）
├── 📋 公司内部文档        ← 预留入口（Phase 3，需求待定）
└── 🎯 营销物料库          ← 预留入口（Phase 4，需求待定）
```

### 4.2 统一内容表设计（content_items）

为了支持未来多种内容类型，数据库采用**统一内容表 + 类型字段**的设计，避免每次新增内容类型就新建一张表：

```sql
CREATE TABLE content_items (
    item_id       TEXT PRIMARY KEY,           -- 唯一ID: MATERIAL-2026-W17-1
    content_type  TEXT NOT NULL,              -- 'moments' / 'learning' / 'internal_doc' / 'marketing'
    sub_type      TEXT,                       -- 子类型：如 'poster', 'case_study', 'policy'

    -- 通用内容字段
    title         TEXT NOT NULL,              -- 标题
    description   TEXT,                       -- 描述/摘要
    body_text     TEXT,                       -- 正文内容（转发语/文档内容/话术）
    body_json     TEXT,                       -- 结构化内容（JSON格式，灵活扩展）

    -- 媒体文件（可选）
    file_path     TEXT,                       -- 主文件路径
    file_name     TEXT,                       -- 原始文件名
    file_size     INTEGER,                    -- 文件大小
    file_type     TEXT,                       -- 'image' / 'pdf' / 'video' / 'doc'
    thumbnail_path TEXT,                      -- 缩略图路径

    -- 元数据
    tags          TEXT,                       -- 标签，逗号分隔（如 "泰国,B2B,收款"）
    week_year     INTEGER,                    -- 年份（用于按周组织的素材）
    week_number   INTEGER,                    -- 周数
    day_of_week   INTEGER,                    -- 周几（1-5）
    publish_date  DATE,                       -- 发布日期

    -- 状态与权限
    status        TEXT DEFAULT 'draft',       -- draft / published / archived
    visibility    TEXT DEFAULT 'all',         -- all / admin_only / channel_only

    -- 统计
    view_count    INTEGER DEFAULT 0,
    download_count INTEGER DEFAULT 0,

    -- 审计
    created_by    INTEGER REFERENCES users(user_id),
    created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_content_type ON content_items(content_type);
CREATE INDEX idx_content_status ON content_items(status);
CREATE INDEX idx_content_week ON content_items(week_year, week_number);
CREATE INDEX idx_content_creator ON content_items(created_by);
CREATE INDEX idx_content_tags ON content_items(tags);
```

### 4.3 当前朋友圈素材专用表（materials）

Phase 1 为了简化实现，仍使用专用表 `materials`，但其字段与 `content_items` 保持兼容。未来迁移时只需写一次数据迁移脚本即可升级到统一表。

**Phase 1 专用表：**

```sql
CREATE TABLE materials (
    material_id   TEXT PRIMARY KEY,           -- 唯一ID: 2026-W17-1
    week_year     INTEGER NOT NULL,           -- 年份: 2026
    week_number   INTEGER NOT NULL,           -- 周数: 17
    day_of_week   INTEGER NOT NULL,           -- 周几: 1-5 (周一到周五)
    publish_date  DATE NOT NULL,              -- 发布日期: 2026-04-27

    -- 内容字段
    theme         TEXT NOT NULL,              -- 主题标签: "泰国B2B收款"
    title         TEXT NOT NULL,              -- 标题: "Ksher泰国B2B收款新升级"
    copy_text     TEXT NOT NULL,              -- 转发语（完整文案）
    copy_short    TEXT,                       -- 短版本转发语（可选）

    -- 媒体文件
    poster_path   TEXT,                       -- 海报原图路径
    poster_name   TEXT,                       -- 原始文件名
    thumbnail_path TEXT,                      -- 缩略图路径（自动生成）
    file_size     INTEGER,                    -- 文件大小(bytes)
    image_width   INTEGER,                    -- 图片宽度
    image_height  INTEGER,                    -- 图片高度

    -- 状态管理
    status        TEXT DEFAULT 'draft',       -- draft(草稿) / published(已发布) / archived(归档)
    scheduled_at  TIMESTAMP,                  -- 定时发布时间（可选）
    published_at  TIMESTAMP,                  -- 实际发布时间

    -- 关联信息
    created_by    INTEGER REFERENCES users(user_id),
    created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    -- 统计数据（实时更新）
    view_count    INTEGER DEFAULT 0,
    download_count INTEGER DEFAULT 0,
    copy_count    INTEGER DEFAULT 0
);

-- 索引
CREATE INDEX idx_materials_week ON materials(week_year, week_number);
CREATE INDEX idx_materials_date ON materials(publish_date);
CREATE INDEX idx_materials_status ON materials(status);
CREATE INDEX idx_materials_creator ON materials(created_by);
```

### 4.2 素材操作日志（material_logs）

```sql
CREATE TABLE material_logs (
    log_id        INTEGER PRIMARY KEY AUTOINCREMENT,
    material_id   TEXT REFERENCES materials(material_id),
    user_id       INTEGER REFERENCES users(user_id),
    action        TEXT NOT NULL,              -- view / download / copy / share
    ip_address    TEXT,
    created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_logs_material ON material_logs(material_id);
CREATE INDEX idx_logs_user ON material_logs(user_id);
CREATE INDEX idx_logs_date ON material_logs(created_at);
```

### 4.3 ER 关系图

```
users (1) ──────< (N) materials         -- 一个用户可以上传多条素材
       │
       └────────< (N) material_logs     -- 一个用户可以操作多条素材

materials (1) ──< (N) material_logs     -- 一条素材可以被多次操作
```

### 5.3 上传表单校验规则

| 字段 | 校验规则 | 错误提示 |
|------|---------|---------|
| 主题标签 | 2-20 字，必填 | "主题标签不能为空" |
| 素材标题 | 5-50 字，必填 | "标题需 5-50 字" |
| 转发语 | 20-500 字，必填 | "转发语需 20-500 字" |
| 海报文件 | PNG/JPG/JPEG，必填 | "请上传海报" |
| 文件大小 | ≤ 5MB | "文件过大，请压缩至 5MB 以内" |
| 图片尺寸 | 建议 1080×1920，不强制 | "建议 1080×1920 以获得最佳效果" |

### 5.4 图片处理流程

```
设计人员上传原图
       ↓
系统自动处理：
  1. 保存原图到 assets/materials/2026-W17-1_original.png
  2. 生成缩略图（等比缩放，最大宽度 400px）
     → assets/materials/2026-W17-1_thumb.jpg
  3. 记录图片尺寸、文件大小到数据库
  4. 如果图片尺寸不是 1080×1920，显示建议提示但不阻止上传
```

### 8.2 缩略图生成

```python
def generate_thumbnail(image_path: str, max_width: int = 400) -> str:
    """
    生成缩略图，保持宽高比，最大宽度 400px
    质量 85%，格式 JPEG
    """
    from PIL import Image

    img = Image.open(image_path)
    ratio = max_width / img.width
    new_size = (max_width, int(img.height * ratio))
    img.thumbnail(new_size, Image.LANCZOS)

    thumb_path = image_path.replace("_original", "_thumb").replace(".png", ".jpg")
    img.save(thumb_path, "JPEG", quality=85)
    return thumb_path
```

## 9. 临时密码保护方案（当前阶段）

### 9.1 实现方式

```python
# config.py
ADMIN_SECRET_KEY = os.getenv("ADMIN_SECRET_KEY", "ksher2026")

# ui/pages/admin/material_upload.py
if not st.session_state.get("admin_authed"):
    password = st.text_input("请输入后台密码", type="password")
    if password == ADMIN_SECRET_KEY:
        st.session_state["admin_authed"] = True
        st.rerun()
    elif password:
        st.error("密码错误")
    st.stop()
```

### 9.2 配置方式

```bash
# .env
ADMIN_SECRET_KEY=your_secure_password_here
```

> 首次部署时，管理员在服务器上设置环境变量密码，然后单独告知设计人员。后续接入正式账号体系后，此临时密码逻辑直接替换为正式登录验证。
