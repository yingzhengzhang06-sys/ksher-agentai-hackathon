# 发朋友圈数字员工 UI/UX 设计

> 文档版本：v1.0  
> 更新日期：2026-04-24  
> 上游文档：`01_MRD.md`、`02_PRD.md`  
> 当前阶段：MVP 高保真交互说明  
> 目标读者：UI/UX 设计师、前端工程师、测试工程师

## 1. 设计目标

页面目标不是展示复杂营销能力，而是让渠道商快速完成一次生成：明确输入、快速提交、清楚理解结果、放心复制。整体风格应贴近现有 Ksher AgentAI 智能工作台，偏工作台、工具化、克制专业，不做营销落地页。

## 2. 设计原则

| 原则 | 说明 |
|---|---|
| 单页完成 | MVP 不跳转，不拆多步向导 |
| 表单优先 | 用户首先看到生成条件，降低认知成本 |
| 结果可扫读 | 正文、合规状态、操作按钮层级明确 |
| 合规不隐藏 | 合规提示紧跟正文，避免用户只复制正文 |
| 移动端可用 | 单列布局、按钮触控面积足够、结果自然换行 |
| 与现有系统一致 | 使用 Streamlit/现有组件风格，不引入独立设计系统 |

## 4. 桌面端布局

推荐宽屏采用左右双栏，窄屏自动变为单列。

| 区域 | 宽屏布局 | 内容 |
|---|---|---|
| 左侧输入区 | 40% 宽度，固定在上方可见 | 所有输入控件和主按钮 |
| 右侧结果区 | 60% 宽度 | 空状态、生成中、结果、反馈 |
| 顶部栏 | 全宽 | 返回、标题、说明 |

宽屏视觉顺序：

```text
┌──────────────────────────────────────────────┐
│ 返回  发朋友圈数字员工                         │
├──────────────────┬───────────────────────────┤
│ 输入条件          │ 生成结果                    │
│ 内容类型          │ 标题/首句                    │
│ 目标客户          │ 正文                         │
│ 产品卖点          │ 合规提示                     │
│ 文案风格          │ 转发建议 / 改写建议           │
│ 补充说明          │ 复制 / 重新生成 / 反馈         │
│ 生成按钮          │                              │
└──────────────────┴───────────────────────────┘
```

## 5. 移动端布局

移动端使用单列布局：

```text
顶部标题
输入区
生成按钮
结果区
结果操作
反馈区
```

移动端规则：

| 项目 | 规则 |
|---|---|
| 表单控件 | 全宽展示，控件间距 12-16px |
| 主按钮 | 全宽，不小于 44px 高度 |
| 多选项 | 允许换行，避免挤压文本 |
| 结果正文 | 按自然段展示，可长按复制 |
| 生成成功 | 自动滚动到结果区顶部 |
| 错误提示 | 滚动到第一个错误字段 |

## 6. 组件设计

### 6.1 页面头部

| 元素 | 类型 | 文案 / 状态 |
|---|---|---|
| 返回 | Icon Button 或文本按钮 | 返回内容工厂 |
| 标题 | H1/H2 | 发朋友圈数字员工 |
| 副标题 | 小号说明 | 生成可复制的朋友圈草稿，发布前请人工确认 |

### 6.2 输入控件

| 字段 | 组件 | 交互 |
|---|---|---|
| 内容类型 | Select | 选项含产品解读、热点借势、客户案例 |
| 目标客户 | Select | 选项含 Amazon、Shopee、外贸 B2B |
| 产品卖点 | Checkbox Group / Multiselect | 最少 1 项，最多 3 项 |
| 文案风格 | Segmented Control / Radio | 专业、轻松、销售感强 |
| 补充说明 | Textarea | 右下角显示 `当前字数/300` |

### 6.3 生成按钮

| 状态 | 表现 |
|---|---|
| 默认 | 主按钮，文案“生成朋友圈内容” |
| 禁用 | 必填项未完成或补充说明超长 |
| Loading | 文案“生成中...”并禁用输入 |
| 失败后 | 恢复可点击，保留用户输入 |

### 6.4 结果卡片

结果区应突出正文，而不是把所有字段做成同等权重。

| 元素 | 展示规则 |
|---|---|
| 标题/首句 | 小标题样式，置于正文上方 |
| 正文 | 主文本区域，字号略大，支持选择复制 |
| 合规状态 | 靠近正文下方，使用状态标签 |
| 转发建议 | 次级信息，说明适合发给谁 |
| 改写建议 | 有风险时高亮，无风险时显示“无”或折叠 |

### 6.5 合规提示视觉

| status | 标签 | 颜色建议 | 行为 |
|---|---|---|---|
| publishable | 可发布 | 绿色或中性色 | 可复制，但仍提示人工确认 |
| rewrite_suggested | 建议改写 | 黄色 | 允许复制，但提示优化 |
| rewrite_required | 需改写 | 红色 | 复制按钮可保留，但需显示警示 |

### 6.6 操作按钮

| 操作 | 按钮样式 | 说明 |
|---|---|---|
| 复制文案 | 主操作按钮 | 只复制 body |
| 重新生成 | 次级按钮 | 保留旧结果直到新结果返回 |
| 有用 | 轻量按钮 | 提交后置为已反馈 |
| 没用 | 轻量按钮 | 展开负反馈表单 |

## 7. 高保真原型描述

### 7.1 首次进入

页面顶部显示标题“发朋友圈数字员工”。左侧输入区控件为空，风格默认“专业”。右侧结果区展示空状态，提示用户填写条件后生成，不展示大面积插画或营销说明。

### 7.2 表单填写

用户选择：

- 内容类型：产品解读
- 目标客户：Amazon 卖家
- 产品卖点：到账快、合规安全
- 文案风格：专业

补充说明为空时也可提交。若没有选择产品卖点，生成按钮可点击但提交后字段下方提示“请至少选择 1 个产品卖点”；也可前端直接禁用，二者选其一保持一致。

### 7.3 生成中

点击生成后：

- 主按钮显示 Loading
- 输入控件暂时禁用
- 结果区显示“正在生成朋友圈草稿”
- 超过 8 秒显示“仍在生成，请稍候”
- 超过 20 秒仍未返回时允许用户重新尝试

### 7.4 成功结果

结果区展示标题、正文、合规提示、转发建议和改写建议。复制按钮位于正文下方，避免用户误复制合规说明。合规状态为 `publishable` 时仍显示“发布前请结合真实业务确认”。

### 7.5 风险结果

当返回 `quality_failed` 或 `rewrite_required`：

- 正文仍展示，便于用户理解问题
- 合规提示置为红色“需改写”
- 改写建议展开显示
- 复制按钮旁增加短提示“建议修改后再发布”

### 7.6 失败兜底

当 AI 超时、返回空内容或格式异常：

- 结果区显示失败提示
- 展示带有“【需要人工补充】”的兜底模板
- 合规状态为“需改写”
- 保留输入，用户可重新生成

## 8. 交互细节

| 交互 | 细节 |
|---|---|
| 复制成功 | Toast：“已复制正文” |
| 复制失败 | Toast：“复制失败，请手动选择正文复制” |
| 重新生成 | 不清空旧结果；新结果返回后替换 |
| 有用反馈 | 点击后禁用该组反馈按钮，显示“已收到反馈” |
| 没用反馈 | 展开原因选择：太空泛、太像广告、不够专业、有合规顾虑、风格不符、其他 |
| 输入变更 | 已有结果保留，但可显示“输入已变化，可重新生成” |

## 9. 文案规范

| 场景 | 文案 |
|---|---|
| 主按钮 | 生成朋友圈内容 |
| 复制按钮 | 复制文案 |
| 重新生成 | 重新生成 |
| 空状态 | 填写左侧条件后，生成一条可复制的朋友圈草稿 |
| Loading | 正在生成朋友圈草稿 |
| 成功 Toast | 已复制正文 |
| 失败 Toast | 生成失败，请稍后重试 |
| 合规提示说明 | AI 生成内容仅作草稿，发布前请人工确认 |

## 10. 可访问性与可用性

| 项目 | 要求 |
|---|---|
| 触控面积 | 移动端按钮和选项高度不小于 44px |
| 文本换行 | 所有长文本自然换行，不遮挡按钮 |
| 错误提示 | 字段级错误在控件附近展示 |
| 键盘操作 | 表单控件可通过 Tab 顺序访问 |
| 色彩 | 不仅依赖颜色表达状态，必须有文字标签 |

## 11. 与现有系统的关系

本页面可作为内容工厂“朋友圈”Tab 中的“单条快速生成”模式，也可独立为 `ui/pages/moments_employee.py`。视觉应复用现有 Streamlit 页面容器、按钮、状态提示和品牌色，避免引入复杂前端依赖。
